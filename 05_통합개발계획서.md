## 1. Executive Summary

### 1.1 프로젝트 개요

소몽의 미국주식 마스터클래스에서 검증된 **3종 판독기(차트/주가/역발상)**를 소프트웨어로 구현하여, 개인 투자자에게 데이터 기반의 체계적 투자 판단 도구를 제공하는 웹 애플리케이션.

### 1.2 핵심 차별화

| 요소 | 기존 서비스(CNN Fear&Greed 등) | 소몽 투자자문봇 |
|------|------|------|
| 분석 체계 | 단일 지표 | 3종 판독기 60점 종합 |
| 적정주가 | 없음 | 소몽 공식 (EPS × 적정PER) |
| 역발상 | 제한적 | 5개 센티먼트 지표 통합 |
| 종목 스크리닝 | 일반 필터 | TOP 20 Tier 분류 |
| 개인화 | 없음 | 포트폴리오 연동 알림 |

### 1.3 기술 스택 결정

```
┌─ 프론트엔드 ──────────────────────────────┐
│  Next.js 15 (App Router) + TypeScript     │
│  Tailwind CSS + shadcn/ui + Recharts      │
├─ 백엔드 ──────────────────────────────────┤
│  Next.js API Routes (Edge/Node Runtime)   │
│  Python 스크립트 (데이터 수집/계산 엔진)    │
├─ 데이터 ──────────────────────────────────┤
│  Supabase (PostgreSQL + Auth + Realtime)  │
│  Upstash Redis (캐싱)                     │
├─ 인프라 ──────────────────────────────────┤
│  Vercel (호스팅 + Cron + Edge Functions)  │
│  GitHub Actions (CI/CD)                   │
├─ 외부 데이터 ─────────────────────────────┤
│  Yahoo Finance (yahoo-finance2)           │
│  FRED API, CBOE (센티먼트)                │
└───────────────────────────────────────────┘
```

## 2. CTO 아키텍처 리뷰

### 2.1 기술 스택 적절성 평가

#### ✅ Next.js 15 — **적절함 (권장)**

| 판단 근거 | 설명 |
|-----------|------|
| SSR/SSG 혼합 | 판독기 결과 페이지는 ISR(4시간 재검증), 대시보드는 CSR |
| API Routes | 별도 백엔드 서버 불필요 → 운영 복잡도 감소 |
| App Router | React Server Components로 초기 로딩 최적화 |
| Vercel 배포 | Zero-config 배포, 글로벌 CDN, Edge Functions |
| 생태계 | shadcn/ui, Recharts 등 금융 UI에 적합한 라이브러리 풍부 |

**⚠️ 주의 사항:**
- API Routes의 실행 시간 제한 (Vercel Hobby: 10초, Pro: 60초)
- 대량 종목 스크리닝(45+종목)은 API Routes가 아닌 별도 Python 파이프라인으로 처리 필요
- `yahoo-finance2` 패키지는 서버사이드에서만 동작 (클라이언트 번들 제외 확인 필요)

#### ✅ Supabase (PostgreSQL) — **적절함 (권장)**

| 판단 근거 | 설명 |
|-----------|------|
| PostgreSQL | 금융 데이터의 정확한 수치 계산 (DECIMAL 타입) |
| RLS | Row Level Security로 사용자별 포트폴리오 격리 |
| Auth | 내장 인증 → 별도 구현 불필요 |
| Realtime | 실시간 알림 기능에 활용 가능 |
| 무료 티어 | 500MB DB, 50K MAU → MVP 충분 |

**⚠️ 주의 사항:**
- 시계열 데이터(일별 주가) 대량 적재 시 파티셔닝 전략 필요
- 무료 티어 DB 크기 제한 → 5년 백테스트 데이터는 별도 스토리지 고려
- Connection pooling 설정 필수 (Supabase는 기본 제공)

#### ✅ Python 데이터 파이프라인 — **적절함 (필수)**

| 판단 근거 | 설명 |
|-----------|------|
| yfinance | 무료 Yahoo Finance 데이터 수집에 사실상 표준 |
| pandas/numpy | 금융 계산(RSI, MA, 이격도)에 최적화 |
| 스케줄링 | Vercel Cron 또는 GitHub Actions로 정기 실행 |

**⚠️ 주의 사항:**
- `yfinance`는 비공식 API → Rate limiting, 데이터 지연 가능성
- Python 스크립트 실행 환경: Vercel은 Node.js 중심 → Python은 별도 실행 필요
- **권장**: GitHub Actions + Supabase 직접 삽입 방식

#### 🔄 아키텍처 개선 권고

```
현재 구조:
  Client → Next.js API → yahoo-finance2 → 실시간 계산 → 응답

권장 구조 (Phase 2):
  ┌─ 데이터 레이어 (비동기) ─────────────────┐
  │ GitHub Actions Cron (4시간마다)           │
  │   → Python collect_data.py              │
  │   → Supabase에 계산 결과 저장            │
  └──────────────────────────────────────────┘
           ↓ (미리 계산된 데이터)
  ┌─ 서빙 레이어 (동기) ────────────────────┐
  │ Client → Next.js API → Supabase 조회    │
  │   → Redis 캐시 확인                     │
  │   → 응답 (< 200ms)                     │
  └──────────────────────────────────────────┘
```

**이유**: Yahoo Finance 실시간 호출은 응답 시간 불안정(2-5초), Rate limit 리스크. 미리 계산/저장 방식이 UX와 안정성 모두 우수.

### 3.2 병목 지점 및 대응 전략

#### 병목 1: Yahoo Finance API Rate Limit

| 문제 | 대응 |
|------|------|
| yfinance 비공식 API 제한 | 데이터 수집은 서버사이드 배치 처리 (4시간 간격) |
| 동시 요청 시 429 에러 | 사용자 요청은 캐시된 DB 데이터 서빙 |
| 1,000명이 동시에 같은 종목 조회 | ISR + Redis 캐시로 Origin 요청 최소화 |

```typescript
// 캐싱 전략 예시
const CACHE_TTL = {
  chartScore: 4 * 60 * 60,     // 4시간 (장중에도 안정적)
  sentimentScore: 1 * 60 * 60, // 1시간 (VIX 변동성 반영)
  fairPrice: 24 * 60 * 60,     // 24시간 (실적 발표 때만 변동)
  portfolio: 5 * 60,           // 5분 (개인화 데이터)
};
```

#### 병목 2: 데이터베이스 부하

| 규모 | 전략 |
|------|------|
| ~1,000명 | Supabase 기본 Connection Pooling (PgBouncer) |
| ~10,000명 | Read Replica 추가, 시계열 데이터 파티셔닝 |
| 인덱스 | `(ticker, date)` 복합 인덱스 필수 |

```sql
-- 시계열 파티셔닝 (10,000명 대비)
CREATE TABLE chart_data (
  id UUID PRIMARY KEY,
  ticker VARCHAR(10),
  date DATE,
  ...
) PARTITION BY RANGE (date);

CREATE TABLE chart_data_2025 PARTITION OF chart_data
  FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');
CREATE TABLE chart_data_2026 PARTITION OF chart_data
  FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');
```

#### 병목 3: 판독기 계산 부하

| 현재 | 개선 |
|------|------|
| 요청 시 실시간 계산 | 배치 계산 → DB 저장 → 조회만 수행 |
| 45종목 × 3판독기 = 135회 API 호출 | Cron Job으로 오프피크 시간에 일괄 처리 |
| 응답 시간 3-5초 | 캐시 히트 시 < 100ms |

```sql
ALTER TABLE portfolios ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can only see own portfolio"
  ON portfolios FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users can only insert own portfolio"
  ON portfolios FOR INSERT

CREATE POLICY "Users can only update own portfolio"
  ON portfolios FOR UPDATE
  USING (auth.uid() = user_id);
```

#### Yahoo Finance (`yahoo-finance2` / `yfinance`)

| 항목 | 세부 |
|------|------|
| 비용 | **무료** (비공식 API) |
| Rate Limit | ~2,000 요청/시간 (추정) |
| 리스크 | Yahoo 정책 변경 시 차단 가능 |
| 대안 | Financial Modeling Prep ($19/월), Alpha Vantage (무료 500/일) |
| 대응 | Phase 1은 무료 API, 유료 전환 시 FMP 이관 준비 |

**비용 시뮬레이션 (배치 처리 기준):**

```
종목 수: 45개 (S&P500 샘플)
판독기: 3종 (차트/주가/역발상)
업데이트 빈도: 4시간마다 (하루 6회)

일일 API 호출 = 45 × 2(차트+재무) × 6 + 1(VIX) × 6 = 546회/일
월간 API 호출 = 546 × 30 = ~16,380회/월

→ Yahoo Finance 무료 범위 내 (2,000/시간 기준, 최대 91회/시간)
```

```typescript
// src/lib/cache/redis.ts
import { Redis } from '@upstash/redis';

const redis = Redis.fromEnv();

export async function getCachedOrFetch<T>(
  key: string,
  fetcher: () => Promise<T>,
  ttlSeconds: number
): Promise<T> {
  const cached = await redis.get<T>(key);
  if (cached) return cached;
  
  const fresh = await fetcher();
  await redis.set(key, fresh, { ex: ttlSeconds });
  return fresh;
}
```

## 12. 부록: 검토 의견 반영

### 12.1 아인슈타인 관점 (강점 강화)

> "Make things as simple as possible, but not simpler."

**적용:**
- 3종 판독기 → 단일 0-60점 통합 점수로 단순화 ✅ (구현 완료)
- 행동 가이드를 자연어로 제공 (코드의 `getActionGuide` 함수) ✅
- 복잡한 내부 계산은 숨기고, 사용자에게는 명확한 시그널만 표시
- **추가 권고**: 게이지 UI에서 "지금 할 일"을 한 문장으로 보여줄 것

### 12.2 소크라테스 관점 (약점 도전)

> "I know that I know nothing."

**도전 질문 및 대응:**

| 질문 | 대응 |
|------|------|
| "판독기 점수가 정말 미래를 예측하는가?" | 예측이 아닌 '확률적 우위' 도구임을 명시. 백테스트 결과 공개 |
| "역발상이 항상 맞는가?" | 역발상 실패 사례도 학습 센터에 포함 (일본 잃어버린 30년 등) |
| "소몽 공식(적정PER=성장률 중앙값)의 한계?" | PER cap 설정 (최대 40), 성장주 별도 로직, 적자 기업 제외 처리 |
| "단일 데이터 소스 의존의 위험?" | DataProvider 추상화로 다중 소스 대비 완료 |
| "사용자가 점수를 맹신하면?" | 교육 콘텐츠 의무 시청, 면책 동의, "참고용" 반복 고지 |

### 12.3 통합 교훈

```
핵심 원칙 3가지:

1. 📊 데이터 겸손: 판독기는 도구이지 예언자가 아니다
   → UI에 항상 "참고 지표"임을 표시
   → 확신도(confidence level) 표시 추가 검토

2. 🔄 기술 유연성: 한 가지 기술에 종속되지 않는다
   → DataProvider 패턴으로 데이터 소스 교체 가능
   → Supabase → 자체 PostgreSQL 이관도 가능한 구조

3. 🛡️ 안전 우선: 금융 데이터는 정확성이 생명이다
   → 이상치 감지 (전일 대비 ±50% 변동 시 경고)
   → 데이터 갱신 시각을 사용자에게 명시
```
