## 제1부: 문제 정의가 정확한가? (55분의 원칙)

### 1.1 ✅ 올바르게 정의된 것

**핵심 문제 인식은 정확하다:**
- "개인 투자자는 정보의 홍수 속에서 뉴스 노이즈에 휘둘린다"
- "체계적 판단 프레임워크 없이 감정적으로 매매한다"
- "소몽의 방법론은 수작업이 번거로워 실행이 어렵다"

소몽 강의의 핵심 메시지("뉴스는 노이즈, 지표로 판단하라")와 PRD의 방향성은 일치한다.

### 1.2 🔴 치명적으로 빠진 문제 정의

#### (1) "누구의 문제"인지가 모호하다

PRD의 페르소나 4종(주린이/직장인/경험자/고수)은 **전부 다른 문제**를 갖고 있다:

| 페르소나 | 실제 문제 | PRD가 해결하려는 것 | 괴리 |
|----------|----------|-------------------|------|
| 주린이 | "뭘 사야 하는지 모른다" | 판독기 점수 제공 | 점수만으로는 행동 불가 |
| 직장인 | "확인할 시간이 없다" | 24/7 모니터링 | 알림은 Phase 2로 밀림 |
| 경험자 | "손절 타이밍을 못 잡는다" | 매매 신호 | 손절 로직 부재 |
| 고수 | "자동화가 필요하다" | 수동 TOP 20 | MVP에선 수동 큐레이션 |

**진단**: 4명 모두에게 "약간 유용하지만 핵심 니즈는 빗나간" 제품이 될 위험이 크다. 아인슈타인이라면 먼저 물을 것이다: **"이 중 정확히 한 명만 고를 수 있다면 누구인가?"**

**권고**: Phase 1에서는 **직장인 투자자** 한 명에 올인할 것. "매일 아침 3분 안에 오늘의 시장 상태와 내 보유종목 상황을 파악" — 이 한 문장이 MVP의 전부여야 한다.

#### (2) 진짜 경쟁자를 직시하지 않았다

이 봇의 실제 경쟁자는 다른 투자 앱이 아니다:

- **경쟁자 A**: 소몽의 네이버 카페/텔레그램 채널 (이미 같은 콘텐츠를 무료 제공)
- **경쟁자 B**: finviz.com + barchart.com + koyfin (3사이트 조합으로 동일 분석 가능)
- **경쟁자 C**: 사용자의 "귀찮음" (소몽도 인정: "어렵지 않은데 번거롭다")

PRD의 위험 테이블에 **"소몽 본인의 무료 채널"**이 경쟁자로 명시되지 않은 것은 가장 큰 문제 정의 실패다.

**권고**: 차별화 포인트를 명확히 할 것. "소몽이 수작업으로 1시간 걸리는 분석을 3초에 자동 산출" — 이것이 유일한 존재 이유.

#### (3) 수익 모델이 부재하다

PRD에 "유료 전환률 5%"가 KPI로 있지만, **무엇을 유료로 할 것인지** 정의가 없다. "투자자문" 명칭 자체가 한국 자본시장법상 **투자자문업 등록 의무**(금융투자업 인가)에 해당할 수 있다.

**권고**: 
- 서비스 명칭을 "투자자문"이 아닌 "투자 분석 도구/교육 보조 도구"로 변경
- 면책조항 강화: "본 서비스는 투자 권유가 아닌 교육 목적의 데이터 분석 도구입니다"
- 금융위원회의 로보어드바이저 테스트베드 규정 검토 필요

## 제2부: 과도한 복잡성은 없는가? (단순화의 원칙)

### 2.1 🔴 판독기 점수 체계의 과잉 설계

**현재 구조**: 3종 판독기 × 4~5개 하위지표 = **총 13개 세부 점수 + 3개 소계 + 1개 총점**

소몽의 원래 엑셀: **4개 숫자 입력 → 1개 종합 점수** (1분 이내)

| 항목 | 소몽 원본 | 현재 구현 | 괴리 |
|------|----------|----------|------|
| 차트 판독기 | 4지표 (MA상회비중200, MA상회비중50, 이격도200, RSI) | MA/이격도/RSI/M·W패턴 | **이동평균선 상회 종목 비중이 빠짐!** |
| 점수 범위 | 1~5점, 평균 산출 | 0~5점, 합산 산출 | 집계 방식 불일치 |
| 시장 vs 종목 | 명확히 분리 | 혼재 | 역할 혼동 |

#### 핵심 발견: 차트 판독기에서 가장 중요한 지표가 누락됨

소몽 강의 원본(2-2강)을 면밀히 분석하면, 소몽이 **가장 강력하게 추천하는 지표**는:

1. **200일 이동평균선 상회 종목 비중** (barchart.com에서 확인, 30%/70% 기준)
2. **50일 이동평균선 상회 종목 비중** (같은 곳, 30%/70% 기준)

이 두 지표가 소몽 차트 판독기의 **핵심 축**이다. 소몽은 이 지표를 "90% 이상 적중률", "거의 예외 없이"라고 반복 강조한다.

그런데 현재 구현의 차트 판독기(`chart-screener.ts`)는 **개별 종목의 MA 위치**(현재가 vs MA50/MA200)를 보고 있지, **시장 전체의 MA 상회 종목 비중**은 전혀 수집하지 않는다.

**이것은 "피아노를 만들었는데 건반을 빠뜨린" 수준의 설계 오류다.**

#### 소몽 원본과의 추가 괴리

| 소몽 원본 | 현재 구현 | 심각도 |
|-----------|----------|--------|
| 이격도는 **200일선** 기준 | 50일선 기준으로 계산 | 🔴 높음 |
| 이격도 구간: -15이하/~-10/~-14/0~10/10~15/15이상 (6단계, 1~5점) | -10이하/~-5/~0/~5/~10/10이상 (6단계, 0~5점) | 🟡 중간 |
| PEG = PE ÷ 성장률(%), **1 이하가 저평가** (피터 린치 공식) | PEG 점수 구간이 0.5/0.75/1.0/1.5/2.0으로 세분화 | 🟡 중간 |
| 적정 PER = 성장률(%) × 1~2배 (강세장 1.5~2.5배) | 적정 PER = 성장률 중앙값 그대로 (1배 고정) | 🔴 높음 |
| 시장 판단과 종목 판단을 **별개 스텝**으로 수행 | 종합 점수에 혼합 | 🔴 높음 |

**권고**: 
1. **시장 판독기**(Market Screener)와 **종목 판독기**(Stock Screener)를 명확히 분리
2. 시장 판독기: MA상회비중(200일+50일) + VIX + 시장PE → "지금 시장은 공포/중립/과열"
3. 종목 판독기: 개별종목 이격도(200일!) + RSI + PE밴드 + PEG → "이 종목은 저평가/적정/고평가"
4. 적정주가 공식에 성장률 배수 범위(×1~2, 강세장 ×1.5~2.5) 반영

### 2.2 🟡 기술 스택의 불필요한 복잡성

**현재**: Next.js + Supabase + Upstash Redis + Python ETL + Vercel Cron

MVP에 이 모든 것이 필요한가?

| 컴포넌트 | 필요성 | 대안 |
|----------|--------|------|
| Supabase (PostgreSQL) | ✅ 필요 | 유지 |
| Upstash Redis | ❌ MVP에 불필요 | Next.js의 revalidate로 충분 |
| Python ETL | ❌ 이중 구현 | TypeScript로 통합 (yahoo-finance2 이미 사용 중) |
| Vercel Cron | 🟡 Phase 2 | MVP에선 수동/API 호출로 대체 |

**진단**: Python(`collect_data.py`)과 TypeScript(`yahoo-finance.ts`)에서 **동일한 로직이 두 번 구현**되어 있다. RSI 계산, 이격도 계산, 점수 산출이 양쪽에 중복. 이것은 유지보수 지옥의 전조.

**권고**: Python ETL 삭제, TypeScript 단일 코드베이스로 통합. "간단한 것이 최고" — 아인슈타인.

### 2.3 🟡 점수 반전(Inversion) 혼동 위험

현재 설계에서 **높은 점수 = 공포 = 매수 기회**라는 역발상 논리가 적용되어 있다. 이것은 소몽의 원본과 일치하지만, **사용자 관점에서 직관에 반한다**:

- "내 종목 점수가 50/60이에요!" → 보통 사용자: "와, 높은 점수! 좋은 거야?" → 실제: "극도 공포 = 시장이 망가져 있다"

PRD의 종합 신호 테이블에서도 혼동이 있다:

```
PRD 종합 신호 (점수 높음 = 공포 = 매수):
0-15: 극도 공포 → 적극 매수
46-60: 극도 환호 → 적극 매도

Combined Screener 코드 (점수 높음 = 공포 = 매수):
≥46: strong_buy (극도 공포)
≤15: strong_sell (극도 환호)
```

코드와 PRD의 방향은 일치하나, **PRD 테이블의 레이블이 역전되어 있다**. PRD에서 "0-15 = 극도 공포 → 적극 매수"라고 했는데, 점수가 낮다 = 지표들이 과열 상태 = 환호 구간이어야 맞다. 코드는 올바르게 구현했으나 PRD 문서의 설명이 반대.

**권고**: 
- 사용자 UI에서는 직관적 스케일 사용 (0=극도과열, 100=극도공포)
- 또는 "공포/탐욕 지수" 형태로 표시 (CNN Fear & Greed Index 패턴 차용)
- 문서 용어 통일 필수

## 제3부: 기존 실패 사례에서 배웠는가? (반복 실수 회피)

### 3.1 🔴 Yahoo Finance 의존의 구조적 위험

**2024년 11월 이후 Yahoo Finance가 rate limiting을 대폭 강화했다:**
- 약 950 티커 이후 429 에러 발생 (yfinance GitHub Issue #2128)
- `yf.Ticker('AAPL').info` 같은 기본 호출도 차단 사례
- 공식 API가 아닌 스크래핑 방식이므로 **언제든 차단 가능**

현재 코드의 위험 지점:

```typescript
// yahoo-finance.ts - 단일 장애점 (Single Point of Failure)
async function getYF() {
  const yf = await import('yahoo-finance2');
  return yf.default; // 이것이 차단되면 서비스 전체 중단
}
```

```python
# collect_data.py - TOP 20 생성 시 45개 종목 순차 호출
for ticker in SP500_SAMPLE:  # 45개 루프 → 높은 rate limit 위험
    chart = collect_chart_data(ticker)
    val = collect_valuation_data(ticker)  # 각 종목당 2회 API 호출
```

**기존 실패 사례**: 수많은 개인 투자 도구가 Yahoo Finance 의존으로 만들어졌다가 API 변경/차단으로 서비스 중단. 이것은 반복되는 실수다.

**권고**:
1. **멀티 소스 전략**: Yahoo Finance를 기본으로 하되, Financial Modeling Prep (FMP)을 fallback으로 구현
2. **캐싱 강화**: 15분이 아닌 **1시간+** 캐싱 (장중 데이터는 어차피 지연)
3. **배치 수집**: 실시간 호출 대신, 하루 1~2회 배치로 전체 종목 데이터 수집 후 DB 저장
4. **에러 처리**: 현재 코드에 `try-catch`와 fallback이 부분적으로만 존재. 체계적 재시도/서킷 브레이커 필요

### 3.2 🔴 센티먼트 데이터의 하드코딩 문제

`sentiment-data.ts`를 보면:

```typescript
// AAII 데이터 - MVP에서는 합리적 기본값 사용
return { bullish: 35, bearish: 30 }; // 기본값 ← 항상 이 값!!

// Margin Debt - Phase 2에서 FINRA 데이터 연동
marginDebtYoY: 0,  // ← 항상 0!!

// Put/Call Ratio - 차단 시 기본값
return 0.85; // 기본값 (중립) ← 실패 시 항상 중립
```

Python 코드(`collect_data.py`)도 동일:

```python
'put_call_ratio': 0.85,  # Phase 2: CBOE 연동
'aaii_bullish': 35,       # Phase 2: AAII 연동
'aaii_bearish': 30,
'margin_debt_yoy': 0,
```

**진단**: 역발상 판독기(20점 만점)의 5개 지표 중 **3개가 하드코딩**이다. VIX만 실제 데이터이고, Put/Call Ratio도 실패 시 기본값. 이것은 판독기가 아니라 **VIX 한 줄짜리 계산기**에 불과하다.

| 지표 | 실제 데이터 | 하드코딩 | 유효성 |
|------|-----------|---------|--------|
| VIX | ✅ | | 유효 |
| Put/Call Ratio | 🟡 시도하나 실패 빈번 | 0.85 | 거의 무효 |
| AAII Bull/Bear | ❌ | 35/30 | 완전 무효 |
| Margin Debt YoY | ❌ | 0 | 완전 무효 |
| HY Spread | 🟡 FRED 의존 | 3.5 | API키 있을 때만 유효 |

**역발상 판독기 실제 유효 점수: 최대 5~10점 / 25점** — 이것은 사용자를 기만하는 것과 다름없다.

**권고**: 
- MVP에서 역발상 판독기를 **2지표로 축소** (VIX + HY Spread만)
- 하드코딩 지표는 "데이터 미수집"으로 명시하고 점수에서 제외
- 또는 역발상 판독기 자체를 Phase 2로 연기하고, MVP는 **차트+주가 2종 판독기**로 출시

### 3.3 🟡 M/W 패턴 감지의 낮은 신뢰도

`chart-screener.ts`의 패턴 감지:

```typescript
function calculatePatternScore(prices: number[]): { score: number; pattern: string } {
  if (prices.length < 20) return { score: 3, pattern: 'neutral' };
  const recent = prices.slice(-20);
  // ... 20일 데이터를 반으로 나눠서 min/max 비교
  // 5% 오차 범위 내 더블바텀/더블탑 판별
}
```

**문제점**:
1. 20일이라는 윈도우가 너무 짧다 (소몽은 주/월간 차트 패턴을 언급)
2. 단순 min/max 비교는 **노이즈에 극도로 민감**
3. W 패턴이 아닌데 W로 판별할 확률이 높다 (false positive)
4. 소몽 강의에서 M/W 패턴은 **차트 판독기의 핵심이 아닌 보조 참고 사항**

소몽의 원본 엑셀에는 M/W 패턴 점수가 **없다**. 차트 판독기는 4개 지표이되: MA상회비중(200일), MA상회비중(50일), 이격도(200일), RSI — 이 4개다.

**권고**: M/W 패턴 감지를 **제거**하고, 대신 MA 상회 종목 비중을 구현할 것.

### 3.4 🟡 적정주가 공식의 한계 미고지

소몽이 강의에서 명시적으로 말한 것:
> "엔비디아 같은 고성장주는 적정주가 판독이 쉽지 않습니다. 이익 사이클이 너무 크기 때문에... 범위가 너무 넓게 나옵니다."

현재 구현에는 이 한계에 대한 **경고 로직이 전혀 없다**:

```typescript
// valuation-screener.ts
const validGrowthRates = data.epsGrowthRates.filter(g => g > 0);
const medianGrowth = validGrowthRates.length > 0 ? median(validGrowthRates) : 15;
// ← 성장률 데이터가 없으면 15%로 가정?!
```

- 성장률 데이터가 0개일 때 15%를 기본값으로 사용하는 것은 **위험한 가정**
- 고성장주(NVDA, TSLA)와 안정주(KO, PG)에 동일한 공식 적용 → 고성장주에서 왜곡

**권고**:
1. 성장률 데이터 신뢰도 레벨 표시 (데이터 0개 → "신뢰도 낮음" 경고)
2. 성장률 변동 계수(CV)가 높은 종목에 대해 "이 종목은 적정주가 산출이 부정확할 수 있습니다" 경고
3. 적정 PER 범위를 성장률 × 1~2배로 수정 (현재 1배 고정은 소몽 공식의 절반만 반영)

## 제4부: 3종 판독기 정확성 및 한계 검증

### 4.1 차트 판독기: 재설계 필요 🔴

**소몽 원본 (2-2강, 2-4강 분석 기반)**:

```
소몽의 차트 판독기 = 4개 지표의 평균(1~5점)

① 200일 이동평균선 상회 종목 비중 (barchart.com)
   - 30% 미만 → 역발상 매수 신호 (1점 = 공포)
   - 70% 이상 → 과열 신호 (5점 = 광기)
   
② 50일 이동평균선 상회 종목 비중 (barchart.com)
   - 동일 기준
   
③ 200일 이격도 (개별 종목 또는 SPY)
   - -15% 이하 → 1점 (극도 공포)
   - +15% 이상 → 5점 (극도 과열)
   
④ RSI (S&P500 또는 개별 종목)
   - 30 이하 → 1점 (과매도 = 매수)
   - 70 이상 → 5점 (과매수 = 매도)

종합 평가 = 4개 점수의 평균
1.0~2.0: 공포 (역발상 매수)
2.0~3.0: 불안 (분할 매수 고민)
3.0~3.5: 중립
3.5~4.0: 과열 (매도 준비)
4.0~5.0: 광기 (분할 매도)
```

**현재 구현과의 차이**:

| 차이점 | 영향 |
|--------|------|
| MA 상회 종목 비중 없음, 개별종목 MA 위치로 대체 | 시장 심리 → 개별 기술로 완전히 변질 |
| 이격도가 50일선 기준 | 소몽은 200일선 기준 명시 |
| 점수 합산(0~20) vs 평균(1~5) | 사소하나 사용자 경험 차이 |
| M/W 패턴 추가 | 소몽 원본에 없는 지표 |

**결론**: 현재 차트 판독기는 소몽의 차트 판독기가 **아니다**. 이름만 같고 내용이 다른 별개의 도구다.

### 4.2 주가 판독기: 부분적 정확 🟡

**소몽 원본 (3-2강, 3-4강 분석 기반)**:

소몽의 주가 판독 방법:
1. **PE 밴드 분석** (koyfin.com에서 확인) — 과거 4~5년 PE 밴드의 고점/저점/평균 파악
2. **적정주가 공식**: 예상 EPS × 적정 PER (적정 PER = 평균 성장률 × 1~2배)
3. **PEG** (yahoo finance에서 확인) — 1 이하: 저평가, 1~1.5: 양호, 2 이상: 고평가

현재 구현의 적정주가 계산:

```typescript
const fairPER = medianGrowth; // 성장률 = 적정 PER (1배 고정)
```

소몽의 실제 공식:

```
적정 PER = 성장률 × 1 ~ 2 (보수적~낙관적)
강세장 적정 PER = 성장률 × 1.5 ~ 2.5
```

즉 현재 구현은 **가장 보수적인 하한값만** 계산하고 있다. 소몽은 범위(레인지)로 제시하는데, 코드는 단일값으로 제시.

또한 PE 밴드 분석(종목별 과거 PE의 고점/저점/평균)이 구현에서 완전히 빠져있다. 이것은 소몽이 "가장 쉬운 방법"이라고 반복 강조한 핵심 분석법.

**권고**: 
1. 적정주가를 범위(보수적~낙관적)로 출력
2. PE 밴드 데이터 수집 및 표시 추가
3. 강세장/약세장 모드 토글 (시장 판독기 결과 연동)

### 4.3 역발상 판독기: MVP에서 사실상 무력 🔴

위 3.2에서 상세 분석한 바와 같이, 5개 지표 중 3개가 하드코딩. **VIX 한 개로 20점짜리 판독기를 운영하는 것은 기만**.

소몽의 역발상 판독기 원본은 강의 챕터 4에서 다루는데, 핵심은:
- 금리(10년물) 환경 판단
- VIX 수준
- 투자자 심리 서베이
- 옵션 시장 심리

이 중 금리 데이터(FRED)와 VIX는 확보 가능하므로, MVP에서는 이 두 개로 축소 운영하는 것이 정직하다.

## 제5부: 소몽 방법론의 AI 최적화 검증

### 5.1 최적화된 부분 ✅

- **지표 계산 자동화**: RSI, 이격도, MA 계산은 AI/알고리즘에 완벽히 적합
- **다중 종목 동시 분석**: 소몽이 수작업으로 6종목 하던 것을 45개+로 확장
- **히스토리 추적**: 과거 점수 변화 트래킹은 엑셀보다 DB가 우월

### 5.2 최적화되지 않은 부분 🔴

#### (1) "판단의 미묘함"이 누락됨

소몽 강의에서 반복되는 핵심 조언:
> "과열이라고 바로 팔지 마시고, 피크를 찍고 **내려오기 시작할 때** 팔아라"
> "30% 밑으로 **내려갔다가 올라오기 시작할 때** 사라"

이것은 **점의 값(point value)**이 아니라 **방향/추세(trend)**의 문제다. 현재 구현은 스냅샷 점수만 산출하고, **점수의 변화 방향**을 추적하지 않는다.

**권고**: 
- `screener_scores` 테이블의 히스토리 데이터를 활용한 **추세 감지** 로직 추가
- "점수 하락 시작" / "점수 상승 시작" 신호 생성
- 이것이야말로 AI가 사람보다 잘 할 수 있는 영역

#### (2) "크로스체크" 로직이 단순 합산

소몽의 접근: "여러 지표를 **크로스체크**해서 확률을 높인다"

현재 구현: 그냥 점수를 더한다 (`totalScore = chart + valuation + sentiment`)

크로스체크는 **가중 검증**이지 **산술 합산**이 아니다. 예를 들어:
- 차트가 "과열"이고 밸류에이션도 "고평가"면 → 매도 확률이 단순 합산보다 높아야 함
- 차트가 "공포"인데 밸류에이션이 "고평가"면 → 혼재 신호, 관망 → 단순 합산은 "중립"으로 처리하는데, 이 경우 "주의" 신호가 더 적절

**권고**: Phase 2에서 가중 교차 검증 로직 도입. MVP에서는 각 판독기 결과를 **독립적으로 표시**하고, 사용자가 직접 크로스체크하도록 유도.

## 제6부: 리스크 관리 로직의 실효성

### 6.1 🔴 손절/리스크 관리가 완전 부재

PRD에 포트폴리오 관리 기능이 있지만, **손절 로직, 최대 손실 한도, 포지션 사이징**에 대한 언급이 전혀 없다.

소몽도 "추격 매수 금지", "현금 비중 확대" 같은 리스크 관리를 강조하는데, 현재 설계는 **"언제 사라"만 있고 "얼마나 사라"와 "언제 손절하라"가 없다**.

**권고**:
1. 매수 신호 시 **비중 권고** 추가 (예: "극도공포 → 전체 투자금의 30% 투입")
2. 포트폴리오 내 종목별 **비중 한도** 경고 (예: 단일 종목 20% 초과 시 경고)
3. 손절 기준: "200일선 이탈 + RSI 30 이하 지속 2주" 등 명시적 기준

### 6.2 🟡 리밸런싱 알림의 구체성 부족

PRD: "월간 리밸런싱 리포트" → 무슨 기준으로? 어떤 로직으로?

**권고**: 리밸런싱 기준을 명시
- 판독기 점수 변화 기반 (예: 차트 점수가 "과열"→"광기" 전환 시 비중 축소 알림)
- 포트폴리오 비중 편향 기반 (예: 기술주 60% 초과 시 분산 권고)

## 제7부: 종합 평가 및 우선순위 권고

### 종합 점수

| 영역 | 평가 | 등급 |
|------|------|------|

| 기술 설계 | 구조는 건전하나 이중 구현/단일 장애점 | C+ |
| 데이터 신뢰성 | Yahoo Finance 의존 + 하드코딩 판독기 | D |
| 리스크 관리 | 거의 부재 | D- |
| MVP 적정성 | 기능 과다, 핵심 미완성 | C |

### 🎯 즉시 수정 필수 사항 (Critical)

1. **차트 판독기에 "MA 상회 종목 비중" 데이터 소스 확보 및 구현**
   - barchart.com 스크래핑 또는 대안 API 확보
   - 이것 없이는 소몽의 차트 판독기라고 부를 수 없음

2. **이격도를 200일선 기준으로 수정**

3. **적정주가 공식을 범위(×1~2)로 수정**

4. **역발상 판독기의 하드코딩 지표를 제거하거나 "미수집"으로 명시**

5. **투자자문 → 투자분석도구로 명칭 변경 + 면책조항 강화**

### 🟡 MVP 전 수정 권장 사항 (Important)

6. Python 코드 삭제, TypeScript 단일 코드베이스
7. M/W 패턴 감지 제거 → MA 상회 종목 비중으로 대체
8. Yahoo Finance fallback (FMP 등) 구현
9. 시장 판독기 vs 종목 판독기 분리
10. 점수 추세 감지 로직 추가 ("피크에서 내려올 때" 감지)
