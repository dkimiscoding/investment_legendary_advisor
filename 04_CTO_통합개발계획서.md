## 1. 아키텍처 리뷰

### 1.1 현행 상태 진단

코드베이스를 리뷰한 결과, **프로토타입 단계**로 평가한다:

| 구성 요소 | 계획 | 현재 구현 상태 | 판정 |
|-----------|------|---------------|------|
| Next.js App Router | ✅ | `next@15.5` + React 19로 구현됨 (PRD의 14가 아닌 15) | ⚠️ 버전 불일치 |
| Yahoo Finance 연동 | ✅ | `yahoo-finance2@3.13` 작동 중 | ✅ |
| 차트 판독기 | ✅ | MA/이격도/RSI/패턴 계산 로직 완성 | ✅ |
| 주가 판독기 | ✅ | 적정주가/PEG 계산 로직 존재 | ✅ |
| 역발상 판독기 | ⚠️ | VIX만 실데이터, 나머지 **하드코딩** | 🔴 |
| Supabase | ❌ | `package.json`에 미설치, 코드에 없음 | 🔴 |
| Upstash Redis | ❌ | 미설치, 캐싱 레이어 없음 | 🔴 |
| 인증(Auth) | ❌ | 미구현 | 🔴 |
| Python ETL | ⚠️ | `collect_data.py` 존재하나 웹앱과 미연결 | 🟡 |
| UI/대시보드 | ⚠️ | 단일 페이지, 기본 게이지 차트 | 🟡 |
| 테스트 | ❌ | 테스트 없음 | 🔴 |

**핵심 발견사항:**
- Python 스크립트와 Next.js 앱이 **이중 구현** — 동일한 계산 로직이 TypeScript와 Python에 각각 존재
- API 응답에 캐싱이 없어 매 요청마다 Yahoo Finance 호출 → **Rate Limit 위험**
- 역발상 판독기의 5개 지표 중 4개가 하드코딩(AAII, P/C Ratio, Margin Debt, HY Spread)

### 1.2 아키텍처 적합성 평가

#### ✅ Next.js + Supabase — 적합하다

이 스택은 **투자 자문 정보 서비스**(실시간 트레이딩 플랫폼이 아닌)에 적합하다:

| 요구사항 | 적합성 | 근거 |
|----------|--------|------|
| 서버사이드 렌더링 | ✅ | SEO + 초기 로딩 성능 (App Router의 RSC) |
| API Route | ✅ | Edge/Node Runtime 선택 가능 |
| 주기적 데이터 업데이트 | ✅ | Vercel Cron (하루 4회면 충분) |
| 사용자 인증 | ✅ | Supabase Auth (OAuth, Magic Link) |
| 데이터 저장 | ✅ | PostgreSQL — 금융 데이터에 적합 |
| 배포/운영 | ✅ | Vercel — 제로 DevOps |

#### ⚠️ 단, 아키텍처 수정이 필요한 부분

**문제 1: Python-TypeScript 이중 구현 해소**

현재 동일한 판독기 계산 로직이 `collect_data.py`와 `src/lib/screeners/`에 중복되어 있다.

```
[현재]
Python(collect_data.py) → 독립 실행 (CLI)
TypeScript(screeners/*) → Next.js API Route (실시간)

[개선안 A — TypeScript 단일화] ← 권장
TypeScript(screeners/*) → Next.js API Route
                        → Vercel Cron (배치)
Python은 폐기. yahoo-finance2 패키지가 충분히 기능함.

[개선안 B — Python 마이크로서비스]
Python(FastAPI) → Railway/Render 배포 → 배치 계산 담당
TypeScript → Next.js → Python API 호출
별도 서버 비용 + 관리 오버헤드 발생. 비추천.
```

**권장: 개선안 A**. 이유:
1. `yahoo-finance2`가 이미 TypeScript에서 잘 작동 중
2. 런타임 통일로 디버깅/배포 단순화
3. Vercel Cron에서 TypeScript로 배치 작업 가능
4. Python은 향후 ML/백테스트 시점에 도입 (Phase 3+)

**문제 2: 캐싱 레이어 부재**

```
[현재] 클라이언트 → API Route → Yahoo Finance (매번 실시간 호출)
[개선] 클라이언트 → API Route → Redis 캐시 확인
                                  ├─ HIT → 캐시 반환 (< 50ms)
                                  └─ MISS → Yahoo Finance → 캐시 저장 → 반환
```

캐싱 전략:

| 데이터 유형 | TTL | 근거 |
|------------|-----|------|
| 차트 데이터 (가격/MA/RSI) | 15분 | 장중 실시간성 확보 |
| 재무 데이터 (EPS/PER) | 24시간 | 분기별 변동 |
| 센티먼트 (VIX) | 5분 | 변동성 지수는 빠르게 변함 |
| 센티먼트 (AAII/Margin) | 7일 | 주간/월간 발표 데이터 |
| TOP20 리스트 | 1시간 | 배치 계산 결과 |
| S&P500 PER | 1시간 | 기준값 |

**문제 3: 데이터 파이프라인 설계**

```
┌─────────────────────────────────────────────────────────────────┐
│                    데이터 파이프라인 (개선)                      │
│                                                                 │
│  [Vercel Cron]──────────────────────────────────┐               │
│  매 4시간 (장중) / 매일 09:00 KST (장전)         │               │
│                                                  ▼               │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐    │
│  │ Yahoo    │   │ FRED     │   │ CBOE     │   │ FMP      │    │
│  │ Finance  │   │ API      │   │ (공개)    │   │ ($19/mo) │    │
│  └────┬─────┘   └────┬─────┘   └────┬─────┘   └────┬─────┘    │
│       │              │              │              │            │
│       ▼              ▼              ▼              ▼            │
│  ┌─────────────────────────────────────────────────────┐       │
│  │              Upstash Redis (캐싱 레이어)              │       │
│  │  차트 데이터 | 재무 데이터 | 센티먼트 | TOP20 캐시    │       │
│  └───────────────────────┬─────────────────────────────┘       │
│                          │                                      │

## 2. 리스크 분석

### 2.1 기술 리스크

#### 🔴 Critical — 즉시 대응 필요

| # | 리스크 | 영향도 | 발생확률 | 대응 방안 |
|---|--------|--------|----------|----------|
| T1 | **Yahoo Finance Rate Limit** | 서비스 중단 | 높음 | Redis 캐싱 + 배치 수집 필수. 무료 티어 제한: ~2,000 req/hr. 500 종목 × 4회/일 = 2,000 호출. 사용자 실시간 요청은 캐시 히트로 처리 |

| # | 리스크 | 영향도 | 발생확률 | 대응 방안 |
|---|--------|--------|----------|----------|
| T4 | **계산 정확도 검증 부재** | 잘못된 투자 신호 | 높음 | 소몽의 실제 판독 결과와 엔진 출력 비교 검증(백테스트). 최소 20개 종목 × 10개 시점 = 200건 수동 검증 필요 |
| T5 | **M/W 패턴 판독 정확도** | 노이즈 시그널 | 높음 | 현행 구현은 20일 데이터 기반 단순 비교. 실제 M/W 패턴은 주간/월간 차트 기반이어야. Phase 2에서 개선하되, MVP에서는 패턴 점수 가중치를 낮추거나 제외 |

| # | 리스크 | 영향도 | 발생확률 | 대응 방안 |
|---|--------|--------|----------|----------|

| T8 | Supabase 무료 티어 제한 | DB 용량 부족 | 낮음 | 500MB DB, 1GB 스토리지. 히스토리 데이터 쌓이면 3-6개월 내 한계. Pro($25/mo) 전환 준비 |

## 3. MVP 재정의

### 3.1 기존 MVP 범위의 문제

PRD의 MVP는 **2개월(8주)**에 3종 판독기 전체 + TOP20 + 포트폴리오를 포함한다. 이는 **1인 개발 기준 비현실적**이다.

특히:
- 역발상 판독기의 5개 지표 중 4개의 데이터 소스가 아직 미구축
- Supabase 연동/Auth/RLS가 전혀 없음
- 캐싱 레이어가 없어 외부 API에 직접 의존

### 3.2 진짜 MVP: 2주 스프린트

> **MVP 목표**: "1개 종목을 입력하면 차트 판독기 점수와 주가 판독기 점수가 나온다. 결과가 캐싱되어 빠르게 반환된다."

#### Week 1: 백엔드 안정화

| 일차 | 작업 | 상세 |
|------|------|------|
| Day 1-2 | **캐싱 레이어 구축** | Upstash Redis 설정, `withCache()` 유틸리티 함수, TTL 전략 적용 |
| Day 3 | **차트 판독기 검증** | 20개 종목 수동 검증, 점수 로직 미세 조정 |
| Day 4 | **주가 판독기 개선** | FMP API 연동(EPS 추정치 정확도 향상), 폴백 로직 |
| Day 5 | **역발상 판독기 (VIX만)** | VIX 단일 지표로 0-20 스케일 재매핑. 나머지는 "준비중" 표시 |

#### Week 2: UI + 배포

| 일차 | 작업 | 상세 |
|------|------|------|
| Day 6-7 | **UI 개선** | 랜딩 페이지, 판독기 결과 카드, 모바일 반응형 |
| Day 8 | **에러 처리 + 로딩 UX** | Skeleton UI, 에러 바운더리, 오프라인 폴백 |
| Day 9 | **Vercel 배포 + 도메인** | 프로덕션 빌드, 커스텀 도메인, 환경 변수 |
| Day 10 | **QA + 소프트 런칭** | 10명 베타 사용자, 피드백 수집 |

#### MVP에서 **제외**하는 것 (의도적 기술 부채)

| 기능 | 이유 | 대체 |
|------|------|------|
| Supabase DB 연동 | MVP는 캐시만으로 충분 | Redis에 결과 캐싱, 히스토리 저장은 Phase 2 |
| 사용자 인증 | 초기엔 로그인 없이 사용 | 무제한 공개 서비스로 시작, Phase 2에서 Auth 추가 |
| 포트폴리오 관리 | Auth 필요, 복잡도 높음 | Phase 3 |
| AAII/Margin Debt/HY Spread | 데이터 소스 구축 시간 | VIX 단일 지표로 역발상 근사. "추가 지표 준비중" 표시 |
| TOP20 리스트 | 500종목 배치 처리 필요 | Phase 2. 수동 큐레이션 5종목으로 시작 |
| M/W 패턴 | 정확도 낮음 | 패턴 점수 0으로 고정, 차트 판독기는 15점 만점으로 운영 |
| Python 스크립트 | 이중 구현 해소 | 삭제. TypeScript 단일화 |
| 테스트 | 속도 우선 | Phase 2에서 핵심 로직 단위 테스트 추가 |

### 3.3 Phase 별 기능 범위 (재정의)

```
┌────────────────────────────────────────────────────────────┐
│  Phase 0: MVP (2주)                                        │
│  ✅ 차트 판독기 (MA + 이격도 + RSI, 패턴 제외)              │
│  ✅ 주가 판독기 (적정주가 + PEG)                            │

### D. 점수 체계 재검토

현행 점수 체계에 대한 CTO 의견:

**문제**: 차트 판독기에서 "높은 점수 = 공포 = 매수 기회"라는 역전 로직이 혼란스러움

| 판독기 | 현행 | 일반적 기대 | CTO 의견 |
|--------|------|------------|----------|
| 차트 | 높을수록 매수 유리 (역발상) | 높을수록 강세 | 현행 유지. 소몽의 철학(공포에 매수) 반영. **단, UI에서 명확히 설명** |
| 주가 | 높을수록 저평가 | 높을수록 저평가 | ✅ 직관적 |
| 역발상 | 높을수록 공포 | 높을수록 공포 | ✅ 직관적 |
| 종합 | 높을수록 매수 기회 | — | ✅ 일관됨 |

**권장**: UI에 "높은 점수 = 매수 기회 (역발상 투자)" 범례를 항상 표시

> **CTO 최종 소견**
> 
> 현재 코드베이스는 **프로토타입으로서 유효**하다. 차트/주가 판독기의 핵심 로직이 작동하고, UI 기반도 갖추어져 있다. 하지만 **프로덕션 수준이 되려면 캐싱, 에러 핸들링, 데이터 소스 다변화가 필수**다.
> 
> 가장 큰 리스크는 Yahoo Finance 의존도와 역발상 판독기의 데이터 부재다. MVP에서 이 둘을 인정하고 축소된 범위로 시작하되, Phase 1에서 반드시 해소해야 한다.
> 
> 비용 구조는 매우 건강하다. 월 $50-80으로 1,000명 규모의 서비스 운영이 가능하며, 유료 사용자 8명이면 손익 분기를 넘긴다. 이것이 이 스택(Vercel + Supabase + Upstash)의 최대 장점이다.
> 
> **첫 2주에 집중할 것**: 캐싱 레이어 구축 + FMP 연동 + UI 개선 + 배포. 그 외 모든 것은 후순위다.
